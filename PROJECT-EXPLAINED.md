# DealTop Project — Full Breakdown

## What Is This Site?

DealTop is an **Amazon Affiliate Site** — it displays products from Amazon and earns a commission on every purchase made through its links.

The site specializes in **electronics** and lists:
- **618** Laptops
- **345** Desktops
- **63** Monitors
- **11** Accessories
- **34** Other products

**Total: 1,071 products**

---

## Where Does the Data Come From?

The data is **not scraped in real time**. The original site ran on WordPress and used a plugin called **WP Automatic** that:

1. **Automatically fetched product data from Amazon** (title, description, images, price)
2. **Sent the product description to GPT** to generate a 1,000-word promotional article
3. Published the article as a WordPress post with an affiliate link

### In the Current Version (Next.js):

The data is **static** — it was extracted from the WordPress `.wpress` backup file and converted to JSON:

```
src/data/products.json    ← All products (1,071)
src/data/pages.json       ← Static pages (Privacy Policy, Terms, etc.)
```

**No scraping happens now** — all data is stored locally in JSON files.

---

## Where Do the Amazon Links Come From?

Every product has an **Amazon Affiliate Link** in this format:

```
https://www.amazon.com/dp/B0F854GHFB?tag=a7medtamer-20
```

### Link Breakdown:
| Part | Meaning |
|------|---------|
| `amazon.com/dp/` | Amazon product page |
| `B0F854GHFB` | Product code (ASIN) on Amazon |
| `?tag=a7medtamer-20` | **Affiliate tag** — ties commissions to the Amazon Associates account |

### Link Source:
- Links were auto-generated by the **WP Automatic** plugin on WordPress
- The plugin grabbed the product ASIN and appended the affiliate tag `a7medtamer-20`
- There are also **10 redirect links** (`/go/1` through `/go/10`) that redirect to specific Amazon products

---

## How the Site Works

1. **All pages are Static (SSG)** — pre-built at build time, not on each request
2. User visits the site → browses products → clicks "View on Amazon"
3. Gets redirected to Amazon with the affiliate tag in the URL
4. If they buy anything on Amazon within 24 hours → **you earn a commission**

# Detailed Comparison: PHP (WordPress) vs Next.js

## 1. Page Rendering

**PHP/WordPress:**
```
User requests a page
  → PHP executes on the server
    → Queries MySQL for data
      → Builds HTML dynamically
        → Sends it to the browser
```
Every time someone opens a page, the server repeats this entire process. That's why the site had **SpeedyCache** and **LiteSpeed Cache** — they stored a pre-built HTML copy so the server wouldn't redo the work every time.

**Next.js:**
```
At build time
  → Reads products.json
    → Generates 1,088 static HTML pages
      → Deploys as static files

User requests a page
  → Receives pre-built HTML instantly (no server, no database)
```

## 2. Database

**PHP/WordPress:**
```
MySQL/MariaDB (11.4)
├── wpak_posts              ← Articles and products (3,314 rows)
├── wpak_postmeta           ← Extra metadata per post
├── wpak_terms              ← Categories/tags
├── wpak_options            ← Site settings
├── wpak_automatic_camps    ← Auto-scraping campaigns
├── wpak_automatic_amazon_links  ← Product-to-Amazon mappings
├── wpak_automatic_log      ← Activity log
├── wc_orders               ← WooCommerce orders
└── 60+ other tables...
```
Everything lived in the database — posts, settings, users, orders, cache, logs.

**Next.js:**
```
JSON files only
├── products.json    ← 1,071 products
└── pages.json       ← 6 static pages
```
**No database at all.** Data is in static files read at build time.

## 3. Automation & Product Fetching

**PHP/WordPress (WP Automatic):**
```php
// Campaign Template stored in the database:

[product_imgs_html]   ← Product images fetched from Amazon automatically

Price: <span style="color:#b12704">[price_with_discount]</span>
(as of [price_update_date] - Details)

<a href="[product_link]"><img src="buy-button.jpg"></a>

[gpt]
Write a detailed promotional product article of around 1000 words based on:
Title: [original_title]
Description: [product_desc]

STRICT RULES:
- Do NOT mention disadvantages
- The article must be 100% positive and promotional
[/gpt]
```

**This happened automatically:**
1. WP Automatic searched Amazon using predefined keywords
2. Fetched product data (ASIN, title, description, images, price)
3. Sent the description to **GPT** to write a 1,000-word promotional article
4. Published the article automatically as a WordPress post
5. Injected the affiliate link (`?tag=a7medtamer-20`)

**Next.js:**
No automation — to add a product you must edit `products.json` manually or build a custom script.

## 4. Plugins vs Code

**WordPress had 10 plugins:**

| Plugin | Function | Next.js Equivalent |
|--------|----------|-------------------|
| **WP Automatic** | Amazon product scraping + GPT articles | None (static data) |
| **Amazon Redirect Manager** | `/go1` - `/go10` redirect links | `src/app/go/[id]/route.ts` |
| **Ad Inserter** | Auto-inject ads into articles | None (can add manually) |
| **WooCommerce** | Store + orders + payments | None (affiliate only) |
| **Featured Image from URL** | External images as featured images | `next/image` with remote patterns |
| **SpeedyCache** | Page caching | Not needed (pages are static) |
| **LiteSpeed Cache** | Advanced caching | Not needed |
| **Classic Editor** | Post editor | None (no admin panel) |
| **Replace Domain** | Domain migration | None |
| **FileOrganizer** | File management | None |

**In Next.js, instead of 10 PHP plugins:**
- `lib/products.ts` — 45 lines of TypeScript handles everything (search, filter, categorize)
- Everything else is built directly into the code

## 5. Security

**PHP/WordPress (.htaccess):**
```apache
# Heavy security rules in .htaccess:
Header always set X-Frame-Options "DENY"
Header always set Content-Security-Policy "frame-ancestors 'none';"
Header always set X-Content-Type-Options "nosniff"
Header always set X-XSS-Protection "1; mode=block"
Header always set X-Robots-Tag "noindex, nofollow"

# Block direct access to affiliate links (no referrer):
RewriteCond %{REQUEST_URI} ^/go[0-9]+/?$
RewriteCond %{HTTP_REFERER} ^$
RewriteRule ^ - [F]    ← Blocked without referrer

# Block external referrers:
RewriteCond %{HTTP_REFERER} !^https?://(www\.)?the-paramedics\.org
RewriteRule ^ - [F]    ← Blocked from other sites
```

**Next.js:**
- No `.htaccess` (no Apache)
- Security comes from having no database and no PHP to exploit
- `robots.ts` blocks search engines from crawling `/go/`
- Referrer protection can be added via `middleware.ts` if needed

## 7. Routing (URLs)

**PHP/WordPress:**
```
the-paramedics.org/                          → index.php → WP_Query
the-paramedics.org/product/laptop-xyz/       → single-product.php → WooCommerce
the-paramedics.org/category/computers/       → archive.php → WP_Query
the-paramedics.org/go1/                      → .htaccess RewriteRule → Amazon
the-paramedics.org/privacy-policy/           → page.php → WP_Query
```
Every URL passes through `.htaccess` first, then PHP decides which template to use.

**Next.js:**
```
/                          → app/page.tsx (Static)
/product/laptop-xyz        → app/product/[slug]/page.tsx (SSG)
/category/laptops          → app/category/[slug]/page.tsx (Dynamic)
/go/1                      → app/go/[id]/route.ts (Server Route)
/privacy-policy            → app/privacy-policy/page.tsx (Static)
/search?q=dell             → app/search/page.tsx (Dynamic)
```
Routing is **file-based** — the file/folder name IS the URL.

## Data Flow: Side by Side

### WordPress (PHP) — How a product page was served:

```
1. User visits: the-paramedics.org/some-laptop-post/

2. Apache reads .htaccess
   ├── Checks security headers (X-Frame, XSS, etc.)
   ├── Checks referrer rules
   └── Routes to index.php

3. WordPress Core (PHP)
   ├── Loads wp-config.php (DB credentials)
   ├── Connects to MySQL
   ├── Runs WP_Query to find the post
   ├── Loads theme template (blogus/single.php)
   └── Executes PHP to build HTML

4. Plugin Pipeline (runs on every page load)
   ├── LiteSpeed Cache → checks for cached copy
   ├── SpeedyCache → secondary cache layer
   ├── Ad Inserter → injects ads into content
   ├── Featured Image from URL → resolves external images
   └── WooCommerce → adds product schema/data

5. Output: Full HTML page → sent to browser
   Total time: 200-800ms (uncached), 50-100ms (cached)
```

### Next.js — How a product page is served:

```
1. At BUILD TIME (once):
   ├── Reads products.json
   ├── For each product, runs product/[slug]/page.tsx
   ├── Generates static HTML + JSON data
   └── Outputs 1,071 HTML files to .next/

2. User visits: the-paramedics.org/product/some-laptop/
   ├── CDN serves pre-built HTML file directly
   ├── Browser receives complete HTML instantly
   └── React hydrates for client-side interactions

   Total time: 10-30ms (always, no cache needed)
```

### WordPress Automation Pipeline (WP Automatic):

```
Cron Job (every X hours)
  │
  ├── Step 1: Search Amazon
  │   └── Keywords: "laptop", "desktop", "monitor"
  │   └── API: Amazon Product Advertising API
  │   └── Returns: ASIN, title, price, images, description
  │
  ├── Step 2: Generate Article
  │   └── Send to GPT: "[original_title] + [product_desc]"
  │   └── Prompt: "Write 1000 words, positive only, no disadvantages"
  │   └── Returns: Promotional article text
  │
  ├── Step 3: Build Post Content
  │   └── Insert product images ([product_imgs_html])
  │   └── Insert price ([price_with_discount])
  │   └── Insert affiliate link ([product_link] + tag=a7medtamer-20)
  │   └── Insert GPT article
  │   └── Insert buy button
  │
  ├── Step 4: Publish
  │   └── Create WordPress post (wp_posts table)
  │   └── Set featured image (wp_postmeta)
  │   └── Assign category (wp_term_relationships)
  │   └── Log activity (wp_automatic_log)
  │
  └── Step 5: Cache
      └── SpeedyCache pre-renders the new page
      └── LiteSpeed invalidates old cache
```

This entire pipeline ran automatically with zero human intervention. In the Next.js version, this automation doesn't exist — all 1,071 products were extracted once from the WordPress database backup.
